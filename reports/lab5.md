# Лабораторная работа №5. Средства управления проектами.

## Ход работы

## 1. Создание и настройка проекта

### 1.1. Создание проекта на GitHub
В репозитории был создан **Проект** по шаблону **Kanban** с названием `Курсовая работа 2026`.

### 1.2. Настройка этапов для задач
Для реализации рабочего процесса были созданы колонки на доске:
- **Backlog** - задачи, которые сформулированы, но не запланированы к выполнению в ближайшее время

- **Ready** - задачи, отобранные для выполнения в ближайшее время, но исполнитель ещё не начал работу

- **In Progress** - задача, над которой исполнитель работает прямо сейчас

- **Review** - задача готова, ожидает проверки преподавателем или согласования

- **Done** - задача принята и закрыта

- **Rework** - задача отправлена на доработку по итогам проверки

### 1.3. Добавление полей в задачи
В настройках созданного проекта были добавлены пользовательские поля:

- **Start date** - тип Date, плановая дата начала работы над задачей

- **End date** - тип Date, плановая дата завершения задачи

- **Тип работы** - тип Single select, возможные значения: Исследовательская работа, Документация, Программирование 

- **Результат** - тип Text, краткое описание того, что должно получиться по итогу выполнения задачи (артефакт)

## 3. Выполнение задачи

### 3.1. Выбор задачи
Для демонстрации процесса выбрана задача: **«Изучение задания и составление постановки задачи»**.

### 3.2. Создание чернового запроса на слияние (Draft PR)
- Ветка `problem-statement` создана от `main` через интерфейс GitHub (кнопка Create branch).
- В эту ветку добавлен файл `курсовая/problem_statement.md` с текстом постановки задачи (требования, цель, ожидаемые результаты).
- Создан **Draft Pull Request** из ветки `problem-statement` в `main` с заголовком **Постановка задачи**.
- В описании PR добавлена ссылка на Issue #13 через команду `Closes #13`, чтобы после слияния PR задача автоматически закрылась.

### 3.3. Привязка PR к задаче
В Issue автоматически появилась ссылка на PR благодаря упоминанию номера Issue. Задача на доске перемещена из **Ready** в **In Progress**.

### 3.4. Движение задачи по доске
| Этап | Действие |
|------|-----------|
| Задача готова, но не начата | **Ready** |
| Начало работы | Перемещена в **In Progress** |
| PR переведён из Draft в обычный | Перемещена в **Review** |

### 3.5. Итоги Review
- Перед тем как отправить задачу на ревью, черновой Pull Request (Draft) будет переведён в обычный с помощью кнопки **Ready for review**.
- Задача на доске будет перемещена в **Rework**, если ее нужно доработать
- Исправления будут внесены в ту же ветку
- После исправлений задача снова будет переведена в **Review**
- PR будет принят и смержен
- Задача будет закрыта и перемещена в **Done**

## 4. Исследование альтернатив

### 4.1. Другие системы управления проектами

Альтернативные инструменты для управления задачами в учебном проекте:

1. **Jira (Atlassian)**

    - **Возможности:**  можно настроить под любой процесс, есть доски, спринты (недельные циклы), отчёты, можно связать с Git.

    - **Ограничения:** бесплатно только для маленькой команды (до 10 человек). Для одного студента это слишком сложная и тяжёлая система.


2. **Trello**

    - **Возможности:** простая доска с колонками, карточки, чек-листы, можно поставить сроки. Есть расширения (Power-Ups), которые добавляют функции.

    - **Ограничения:** бесплатно только 1 Power-Up на доску, платный тариф от $5/мес.


3. **YouTrack (JetBrains)**

    - **Возможности:** доски для гибких методологий (Agile), можно настроить под себя. Для маленьких команд бесплатно.

    - **Ограничения:** бесплатно до 10 человек. Настраивать сложнее, чем Trello или GitHub Projects.


4. **Asana**

    - **Возможности:** задачи и доски, календари, зависимости между задачами.

    - **Ограничения:** бесплатная версия только с базовыми возможностями. Чтобы получить больше, нужно платить от $11 в месяц.


5.  **GitLab Issues**

    - **Возможности:** встроен в GitLab, аналоги GitHub Issues, доски, вехи(этапы проекта), можно встроить автоматические проверки кода (CI/CD).

    - **Ограничения:** бесплатно, но есть ограничение по размеру репозитория.

### 4.2. Выбранная альтернатива: Trello

Для сравнения выбрана система **Trello** как наиболее близкая по простоте к GitHub Projects и доступная в бесплатной версии для учебных целей.

### 4.3. Как можно выполнить те же задания, используя Trello

| Что нужно было сделать | Как это сделать в Trello |
|-----------------------|--------------------------|
| Создать проект и дать доступ преподавателю | Создать доску «Курсовая работа 2026». Добавить преподавателя как **Member** по email |
| Настроить колонки для задач | Создать списки: **Backlog**, **Ready**, **In Progress**, **Review**, **Done**, **Rework** |
| Добавить даты начала и окончания | Использовать Power-Up **Custom Fields**. Добавить поля типа **Date** |
| Добавить тип работы | В **Custom Fields** добавить поле типа **Dropdown** с вариантами: Исследовательская работа, Документация, Программирование  |
| Разбить работу на маленькие задачи | Создать **cards** для каждой задачи |
| Обозначить, что одна задача зависит от другой | В описании карточки написать вручную: **blocked by card #N** (автоматических зависимостей в бесплатной версии нет) |
| Создать черновик запроса на слияние (Draft PR) | В Trello нет функции **Pull Request**. Код хранится на GitHub, а в Trello только карточка. Создать карточку с пометкой **DRAFT** в названии и перемещать её по колонкам |
| Передвигать задачу по колонкам | Перетаскивать карточку мышкой между списками |
| Перевести черновик в готовый | Убрать из названия карточки слово **DRAFT** и переместить её в колонку **Review** |

### 4.4. Основные ограничения Trello для этой лабораторной работы

- Нет встроенной связи с **Pull Request** (как в GitHub)
- Поля с датами и типами работ требуют Power-Up **Custom Fields** (бесплатно — только один Power-Up на доску)
- Задачи не двигаются автоматически при открытии/закрытии PR — всё вручную
- Связи типа **blocked by** работают только через ручную надпись в описании
